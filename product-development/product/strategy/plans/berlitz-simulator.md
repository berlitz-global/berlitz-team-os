# Berlitz Simulator — Plan

**Created:** 2026-06-14
**Status:** Planned
**Owner:** Jan Hoffmann

## Context

Financial data currently lives in three disconnected formats: an Excel cost model (v2.6), a hand-coded HTML dashboard (benchmarks v3), and markdown tables in business-info.md. The same numbers appear in all three but are maintained independently — when one changes, the others don't. AI agents can't read the Excel, and the HTML has every value hardcoded.

The Berlitz Simulator creates a single canonical data layer (YAML) that bridges the Excel model (where Jan works) with the rest of team-os (where AI agents, dashboards, and team members consume data). It follows team-os principles: CLAUDE.md indexes, structured templates, git-versioned, AI-readable.

## Approach: Phased — Excel now, YAML bridge, notebook later

```
Phase 1:  Excel ──(you update)──→ YAML ──→ AI agents, markdown summaries
Phase 2:  Excel ──(you update)──→ YAML ──(script)──→ HTML dashboards
Phase 3:  Notebook ──(generates)──→ YAML ──(script)──→ HTML dashboards
```

YAML is the stable contract. The engine behind it can change.

## File details

### 1. inputs/assumptions.yaml

The single source of truth. Every number in the model traces back here. ~100 lines. Three groups matching the "3 digestible categories" requirement:

- **Group 1 — Pricing & Revenue**: tier prices, credits, session allowances per tier
- **Group 2 — Cost Drivers**: AI per-session costs (LLM, STT, TTS, pronunciation, avatar), human teacher rates (LATAM/EU), platform overhead
- **Group 3 — Funnel & Retention**: signups, conversion rate, tier distribution, churn

Every value carries:
- `status`: `assumption` | `estimate` | `actual` — makes it visually obvious what's proven vs guessed
- `source`: where the number comes from (Excel cell, board deck, production data)
- `updated`: when last verified

### 2. inputs/actuals.yaml

Starts empty. As the mobile app launches (Q4 2026), measured values go here. The convention: actuals override assumptions. Each entry records the measured value, measurement date, sample size, and what assumption it replaced.

### 3. inputs/scenarios.yaml

Named scenarios that override specific assumptions. Start with 3-4:
- `mobile_launch_india` — India pricing ($10.80), adjusted conversion
- `bear_case` — low conversion (10%), high churn (10%)
- `bull_case` — high conversion (30%), low churn (4%)
- `no_avatar` — what if we drop the avatar from AI Tutor

### 4. model/formulas.md

Plain-English documentation of every formula. Example:
- "Plus gross profit = Plus revenue - Plus AI cost - Plus platform cost"
- "Plus AI cost = users x (practice_sessions x practice_cost + tutor_sessions x tutor_cost)"

This lets anyone verify calculations by hand against the Excel.

### 5. simulator/CLAUDE.md

Tells AI agents:
- Where to find current assumptions: `inputs/assumptions.yaml`
- Where to find results: `outputs/summary.md`
- That `business-context/berlitz_cost_model_v2.6.xlsx` is the Excel engine, not the canonical data source
- How to run a scenario: "read scenarios.yaml, apply overrides to assumptions.yaml, compute using formulas.md"

### 6. outputs/summary.md

A clean markdown table showing unit economics per 1,000 signups, per-tier breakdown, and aggregate metrics. Can be generated by Claude on demand or by a script later.

### 7. changelog.md

Records major assumption changes with dates and rationale. Example: "2026-06-14: Avatar cost updated from $0.45/min to $0.03/min — Simli volume pricing quote received."

## Single source of truth mechanisms

1. **YAML is canonical for team-os**: All AI agents, dashboards, and documents reference assumptions.yaml. Excel is the "engine room" where Jan works, but YAML is the "published API."

2. **Status tags create transparency**: Every value in assumptions.yaml is tagged `assumption`, `estimate`, or `actual`. Anyone can see at a glance how much of the model is proven vs guessed.

3. **Deprecation headers on old artifacts**: Add a 1-line note to `berlitz-business-info.md` and the HTML dashboard pointing to `simulator/inputs/assumptions.yaml` as the canonical source.

4. **Update convention**: When you change a number in Excel, update the matching line in YAML. This is a 2-minute task — you're updating ~1 line, not duplicating the model.

5. **Git history as audit trail**: Every YAML change is a commit with context. No separate versioning needed.
